# 同步逻辑、规则、冲突处理

## 一、三路持久化

```
                 save()
                   │
     ┌─────────────┼─────────────┐
     ▼             ▼             ▼
 localStorage   本地 JSON    GitHub Gist
  （浏览器缓存）  （绑定文件）   （云端）
```

| | 写 | 读（页面加载） | 性质 |
|---|---|---|---|
| **localStorage** | 每次 save() 同步写 | load() 主数据源 | 主存储，永远有 |
| **本地 JSON** | save() 后 600ms 防抖写 | ❌ 不读回 | 单向备份（防清缓存丢数据） |
| **Gist** | save() 后 2s 走 syncCycle | 拉取后 mergeState 合并 | 多端同步 |

## 二、何时同步

| 事件 | 触发 | 行为 |
|---|---|---|
| 改词条/加词/删词 | save() → scheduleCloudSync() | 2s 防抖后走 **syncCycle**（拉→合并→推） |
| 打开页面 | startCloudSync() | **syncCycle**（拉→合并→推） |
| 切回页面（visibilitychange） | pullFromGist | 只拉取合并（不推） |
| 点「🔄 立即同步」 | syncNow() → syncCycle(true) | 拉→合并→推 |
| Python 直改 JSON 文件 | ❌ 不触发 | 需手动点🔄同步 |

## 三、核心规则：push 前必须先 pull+merge

⚠ **这是修复后的关键规则。** 之前的 bug：`scheduleCloudSync` 只调 `pushToGist`（裸推），不先拉取。
导致手机端本地 12 条旧数据直接覆盖了云端 85 条。

**修复后**：所有推送都走 `syncCycle` = 原子的「拉取→合并→推送」：

```
syncCycle():
  1. pullFromGist(silentMerge)   拉取云端 + mergeState 合并到本地
  2. pushToGist()                推送合并后的本地数据
  （_syncCycleRunning 锁防止并发）
```

**绝不再裸 push**。推送的数据一定是「本地 ∪ 云端」合并后的全集。

## 四、合并算法 mergeState

```js
mergeState(local, remote):
  byId = Map()
  // 本地 + 远程所有条目按 id 合并
  for e in [...local.entries, ...remote.entries]:
    if byId 没有 e.id 或 e.updatedAt 更大:
      byId[e.id] = e        // 同 id 取 updatedAt 大的（后改的赢）

  // 墓碑：删除记录
  for t in [...local.tombstones, ...remote.tombstones]:
    byId.delete(t.id)       // 已删除的条目从结果移除

  return { entries, tombstones, tagGraph(取 updatedAt 大的) }
```

## 五、增量 vs 全量

- **传输**：Gist 每次 push 是**全量覆盖**（整个 JSON 文件替换）。GitHub Gist API 不支持增量。
- **合并**：客户端 `mergeState` 是**按 id 增量合并**（取并集，同 id 取新）。

所以：小数据集永远不会覆盖大数据集——合并取并集。

## 六、冲突处理

### 场景1：两端都改了同一条词条（字段级）

```
桌面: helado updatedAt=100  person=""
手机: helado updatedAt=200  person="第三人称"
```
→ **updatedAt 大的赢**（手机版，含 person）。整条替换，不按字段合并。

### 场景2：一端删、一端改

```
桌面: 删了 id5 → 墓碑 {id5, deletedAt=300}
手机: 改了 id5 → updatedAt=200
```
→ **墓碑赢**（删除优先）。id5 被删。

### 场景3：两端各自新增不同词条

```
桌面: 新增 id100
手机: 新增 id101
```
→ **都保留**（并集）。合并后两条都在。

### 场景4：Python 直改 JSON 未更新 updatedAt

```
本地: helado person="X" 但 updatedAt=100（旧）
云端: helado updatedAt=200（无 person）
```
→ **云端赢，person 丢失** ❌

**对策**：Python 改数据后必须把 updatedAt 刷为当前时间，否则合并时会被旧版覆盖。

## 七、删除同步

删词条 → `deleteEntry` 产生墓碑 `{id, deletedAt}` → 推到云端 → 其他端拉取 → mergeState 应用墓碑 → 该端也删除。

墓碑永久保留（防止已删条目被旧端复活）。

## 八、时间戳 updatedAt

- 每次通过页面改词条 → `saveEntry` 自动设 `updatedAt = Date.now()`
- Python 直改 → **必须手动设** `updatedAt = 当前毫秒时间戳`
- 合并时 `updatedAt` 大的赢

## 九、状态栏（首页顶部常驻）

```
💾85 · ☁️85 · ↑18:59 · ↓19:00 · 📁✓
```

| 符号 | 含义 |
|---|---|
| 💾N | 本地 localStorage 条数 |
| ☁️N | 云端 Gist 条数（? = 未拉取过） |
| ↑HH:MM | 最后上传时间 |
| ↓HH:MM | 最后拉取时间 |
| ⟳ | 同步进行中（闪烁） |
| ⚠ | 上次同步出错（悬停看详情） |
| 📁✓ | 本地文件已绑定 |

## 十、缺陷修复记录

| 缺陷 | 现象 | 修复 |
|---|---|---|
| 裸 push 覆盖 | 手机 12 条覆盖云端 85 条 | push 前强制 pull+merge（syncCycle） |
| Python 改数据未刷 updatedAt | 新字段被旧版覆盖 | 手动刷 updatedAt |
| 双 PATCH 冲突 | 第一次写签名冲掉真实数据 | 移除冗余 PATCH |
| busy 锁阻止 syncCycle 内 pull | pullFromGist 检查 syncStatus.busy | silentMerge 模式跳过 busy 检查 |
