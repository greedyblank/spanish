# 单击/双击交互语义 + 三系统并行（highlight 与 filter 彻底解耦）

- 日期：2026-07-10   状态：已采纳

## 背景

原交互把「高亮/联动」和「筛选/分组」混在一起，用户体感混乱：

- 单击左栏词根 → 竟然会 `state.filter.letter = 词根首字母`，把词条列表**过滤**成该字母下全部词条。用户只想看这个词根的信息，却被动进入了字母分组。
- 双击词条 → 同时设 filter.letter + highlightRoot，也进入筛选。
- 搜索 / 标签 / 字母互相清空：选字母清空标签和搜索，搜索清空字母和标签——三者**互斥**，无法叠加使用。

用户明确要求：
1. 单击词源/词条 = 只高亮 + 详情，**不筛选、不过滤、不进入分组**。
2. 双击 = 联动跳转（词源→第一个单词；词条→它的词源），**也不筛选**。
3. 只有「单击导图字母」或「单击词根分组的分组标题」才有筛选逻辑。
4. 词条→词根反向联动时，若可关联多个词根，默认反向关联到主词根。
5. 标签系统 / 词根词缀系统 / 搜索系统要**并行不悖**。

## 抉择

- **选项 A**：小修——单击词根时不再设 filter.letter，但保留搜索/标签/字母互斥。只解决部分问题。
- **选项 B**：彻底把 `highlightRoot`（高亮/详情维度）与 `filter`（筛选维度）解耦，并把搜索/字母/标签重构成**可叠加求交**的三个独立维度。

## 决定

采用 **B**。

**两类状态彻底分离：**
- 高亮/选择（只影响详情面板 + 列表高亮）：`highlightRoot`、`selectedEntryId`。
- 筛选（只影响词条列表范围）：`filter.letter` + `filter.search` + `tagCloudState.selectedTag`，三者**并行求交**。

**交互映射：**
| 操作 | 行为 |
|---|---|
| 单击词源 `selectRoot` | 切换 highlightRoot，详情切到词根详情。**不动 filter** |
| 双击词源 `dblClickRoot` | 选中该词根第一个单词（高亮+词条详情）。不筛选 |
| 单击词条 `selectEntry` | 选中词条，清 highlightRoot。不筛选 |
| 双击词条 `dblClickEntry` | 反向到主词根 `getRootKey(e)`（多词根默认主词根），详情切词根详情。不筛选 |
| 单击导图字母 `selectLetter` | 切换 `filter.letter`（**唯一**筛选入口之一），保留 tag/search |
| 单击词根分组字母标题 | `selectLetter(L)`，同上 |

**三系统并行**（`getVisibleEntries()`）：`搜索结果 ∩ 字母 ∩ 标签子树`，任一维度都不清空其它维度；标题用 `∩` 拼接展示当前生效的筛选。

**多词根反向联动**：`dblClickEntry` 用 `getRootKey(e)`（主词根）作为默认反向目标；详情卡词根区把全部 `getRootKeys(e)` 渲染为可点击 chip，用户可主动跳到附加词根。

## 后果

- 单击一个词根不再"跳走"，符合"我只是想看看它"的直觉。
- 可以「H 字母 + 饮食标签 + 搜索 hel」三条件叠加，真正并行。
- `render()` 中列表渲染统一走 `getVisibleEntries()`，`renderWordsList`/`renderRootsList`/搜索分支不再各写一套过滤，减少分歧。
- **词根/词缀入词表首条**（用户 #2）落在词根详情的「同族词」区：`getEntriesForRoot` 把词根/词缀自身条目排最前，并加「本词根/本词缀」标记。
- 与韩/日姊妹项目：此交互模型更清晰，建议反向同步到韩/日（它们也有类似 highlight/filter 混淆的历史包袱）。
